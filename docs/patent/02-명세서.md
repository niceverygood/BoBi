# 명세서

**【발명의 명칭】**
AI 기반 보험 미래 시나리오 통합 비교 방법 및 시스템
(METHOD AND SYSTEM FOR INTEGRATED COMPARISON OF FUTURE INSURANCE SCENARIOS USING ARTIFICIAL INTELLIGENCE)

---

## 【기술분야】

본 발명은 인공지능 기반 보험 상담 지원 시스템에 관한 것으로, 보다 구체적으로는 피보험자의 의료이력 데이터를 기반으로 장래 질병 위험도를 예측하고, 해당 질병 발병 시 예상 의료비를 산출하며, 현 시점 및 미래 시점에서의 보험 가입 가능성을 추정하여, 복수의 보험 가입 시나리오별 자기부담금과 보장 공백을 통합 비교·제시하는 컴퓨터 구현 방법 및 시스템에 관한 것이다.

---

## 【발명의 배경이 되는 기술】

[0001] 보험설계사가 고객에게 보험 상품을 설계·안내함에 있어, 종래에는 (i) 고객의 과거 진료 이력 정리, (ii) 가입 가능 보험상품 매칭, (iii) 예상 의료비 계산, (iv) 미래 건강 위험 예측이 각각 분리된 도구와 수작업으로 이루어져 왔다.

[0002] 대한민국 국민은 건강보험심사평가원(이하 "심평원") 및 국민건강보험공단(이하 "건보공단")으로부터 자신의 진료 이력 데이터를 발급받을 수 있으며, 이를 PDF 또는 Open API(CODEF 등)를 통해 확보할 수 있다. 그러나 이러한 원천 데이터만으로는 보험 가입 실무에서 요구되는 고지사항 분류, 가입 가능 상품 판정, 미래 질병 위험도 추정, 보험금 대비 자기부담금 시뮬레이션을 수행하기 어렵다.

[0003] 또한 종래의 보험 상담에서는 "현재 가입하는 경우"와 "향후 일정 기간 후 가입하는 경우"를 직접적·정량적으로 비교하여 고객에게 제시하는 수단이 존재하지 아니하여, 가입 의사결정이 설계사의 구두 설명에 크게 의존하는 한계가 있었다.

[0004] 특히, 보험 인수(underwriting) 심사는 계약자의 과거 질병 이력뿐 아니라 현재 만성질환·복용 약물·향후 발병 가능성을 종합하여 판단되는데, 이를 설계사 단에서 5년 등 장래 시점에 대해 정량 추정하는 도구는 공지되지 아니하였다.

## 【발명의 내용】

### 【해결하고자 하는 과제】

[0005] 본 발명은 다음의 과제를 해결하는 것을 목적으로 한다:

(1) 피보험자의 의료이력(진료 기록, 복용 약물, 건강검진 결과)을 단일 입력으로 수용하여, 장래 질병 위험도 예측과 질병별 예상 의료비 및 장래 시점 보험 인수 가능성 추정을 **하나의 오케스트레이션**으로 수행하는 방법을 제공한다.

(2) 현재 가입 시나리오, 장래 N년(예: 5년) 후 가입 시나리오, 미가입(현상유지) 시나리오의 **세 가지 시나리오별 자기부담금을 정량 산출**하여 설계사 및 고객이 동일 지면에서 비교할 수 있도록 한다.

(3) 장래 시점의 인수 가능성을 **현재 진료 이력의 가중치 합산 + 장래 위험 질환의 카테고리 가중치 + 복용 약물 개수 가산 + 위험 플래그 점수화**라는 복수의 피쳐를 정규화된 위험 점수로 산출하고, 이를 3단계 outcome 및 보험료 멀티플라이어·예상 면책항목으로 변환하여 제공한다.

(4) 전술한 시나리오 비교 결과를 **제3자(고객)에게 안전하게 전달**하기 위한 HMAC 서명 기반 토큰화된 공유 URL 및 알림톡 연계 수단을 제공한다.

### 【과제의 해결 수단】

[0006] 전술한 과제를 해결하기 위하여 본 발명에 따른 AI 기반 보험 미래 시나리오 통합 비교 방법은 다음의 단계를 포함한다:

(a) **의료이력 수신 단계**: 컴퓨터 시스템이 피보험자로부터 전자문서(PDF) 또는 의료정보 조회 API(예: CODEF)를 통해 진료 기록 및 복용 약물 데이터를 수신하는 단계;

(b) **위험도 리포트 생성 단계**: 상기 의료이력을 사전 구축된 질환-후속질환 매핑 테이블과 매칭하여 매칭된 위험 질환 목록을 추출하고, 대규모 언어모델(LLM)에 상기 의료이력 요약 및 매칭 결과를 입력하여 장래 발생 위험이 있는 질환 목록·상대위험도·위험 수준·근거를 포함하는 위험도 리포트를 생성하는 단계;

(c) **질병-비용 매칭 단계**: 상기 위험도 리포트의 위험 질환 상위 N개를 기준으로, 사전 구축된 질환별 평균 진료비 데이터베이스(급여·비급여·본인부담비율·평균 투병기간을 포함) 및 질환별 치료과정·비용근거·생활영향 데이터베이스와 매칭하는 단계;

(d) **5년 후 인수 가능성 추정 단계**: 상기 의료이력의 현재 진단코드에 기 설정된 가중치를 부여하고, 상기 위험도 리포트 상위 위험 질환에 카테고리별 가중치·위험수준 배율을 적용하며, 복용 약물 개수에 따라 가산점을 부여하고, 의료이력 내 위험 플래그의 심각도별 점수를 합산하여, 0 내지 100으로 정규화된 인수거절 위험 점수(declineRiskScore)를 산출하는 단계;

(e) **시나리오 산출 단계**: 설계사로부터 (i) 제안 보험금(proposedInsurancePayout), (ii) 추가 월 보험료(additionalMonthlyPremium), (iii) 고객 월 생활비(monthlyLivingCost), (iv) 현재 가입 보험금(currentInsurancePayout)을 입력받아,
  - **시나리오 A(지금 보완)**: 현재 보험금 + 제안 보험금을 보장으로 하여 자기부담금 = max(0, 본인부담금 + 생활비손실 - 보장)을 산출하고,
  - **시나리오 B(N년 후 가입)**: 상기 인수 가능성 추정 결과(outcome)에 대응하는 보장률(예: 인수가능 85%, 조건부 50%, 인수거절 10%)을 제안 보험금에 적용하고, 예상 면책 항목과 매칭된 질병은 추가로 보장률을 차감하여 자기부담금을 산출하며, 월 보험료는 상기 인수 가능성 점수에 따른 멀티플라이어를 현재 추가 월 보험료에 곱하여 산출하고,
  - **시나리오 C(미조치)**: 현재 보험금만을 보장으로 하여 자기부담금을 산출하는 단계;

(f) **시나리오 설명 생성 단계**: LLM에 상기 위험도 리포트 요약, 인수 추정 결과, 시나리오별 금액을 포함하는 프롬프트를 전달하여, 시나리오 A, B, C 각각에 대한 상담용 자연어 메시지 및 핵심 요약(keyTakeaways)을 생성하는 단계;

(g) **결과 통합 출력 단계**: 상기 위험도 리포트, 질병별 시뮬레이션, 3가지 시나리오 비교, 인수 추정을 포함하는 통합 리포트를 설계사 단말에 렌더링하는 단계.

[0007] 본 발명에 따른 방법은 추가로, 상기 통합 리포트에 대응하는 HMAC 서명 기반 토큰을 발급하고, 상기 토큰을 파라미터로 포함하는 공유 URL을 생성하며, 상기 URL을 버튼 링크로 갖는 알림톡을 피보험자의 이동통신 단말로 발송하는 단계를 더 포함할 수 있다.

[0008] 본 발명에 따른 방법은 추가로, 상기 LLM에 전달되는 프롬프트 내에 피보험자별 상위 위험 질환의 진료 이력 근거(sourceDisease, sourceCode, explanation)를 구조화하여 주입함으로써, 생성되는 시나리오 설명이 해당 피보험자의 개별 병력에 근거한 자연어 설명(whyRiskyForCustomer, riskInPlainLanguage)을 포함하도록 하는 단계를 포함할 수 있다.

### 【발명의 효과】

[0009] 본 발명에 의하면 다음과 같은 효과가 있다:

(1) 의료이력 원천 데이터 수신부터 3시나리오 비교 출력까지를 하나의 자동화된 파이프라인으로 통합함으로써, 종래 수작업 기반 상담에서 요구되던 수십 분의 수작업을 수 초 내지 수십 초의 자동 처리로 단축한다.

(2) 현재 시점의 가입 의사결정이 미래 시점의 건강·인수 상황에 미치는 영향을 **정량적·시나리오별로 동시 제시**하여, 고객의 가입 의사결정 합리성을 높인다.

(3) 장래 시점 인수 가능성을 **개별 피보험자의 진료 데이터**에 기반하여 추정함으로써, 일반 통계에 기반한 종래의 상담보다 개인화된 의사결정 근거를 제공한다.

(4) HMAC 토큰 기반 공유 및 알림톡 연계를 통하여, 상담 현장에서 즉시 리포트를 고객에게 전송할 수 있고, 제3자 유출을 방지하는 보안성을 동시에 확보한다.

## 【발명을 실시하기 위한 구체적인 내용】

### 1. 시스템 구성 (도 1, 도 8 참조)

[0010] 본 발명에 따른 시스템(100)은 다음 구성요소를 포함한다:

- **설계사 클라이언트(110)**: 웹 브라우저 또는 모바일 애플리케이션 형태로, 설계사가 고객 데이터를 입력하고 리포트를 열람하는 단말.
- **애플리케이션 서버(120)**: Next.js 기반 App Router 구조에서 동작하며, API 라우트와 서버 컴포넌트를 통해 비즈니스 로직을 수행한다.
- **데이터베이스(130)**: PostgreSQL(예: Supabase)로 구현되며, 분석 기록(analyses), 고객(customers), 구독(subscriptions), 크레딧(credit_balances), 사용량(usage_tracking) 등의 테이블을 포함한다.
- **인공지능 서비스(140)**: 외부 LLM(예: Claude Sonnet 4.5) 또는 LLM 라우팅 게이트웨이(예: OpenRouter)를 통해 호출되며, 프롬프트 입력 및 JSON 응답을 주고받는다.
- **의료정보 API 어댑터(150)**: 심평원/건보공단 데이터 스크래핑 API(예: CODEF)를 호출하여 진료·약·건강검진 데이터를 수집한다.
- **PDF 파서(160)**: `pdf-parse`, `pdfjs-dist` 등을 사용하여 PDF 원천 문서에서 텍스트를 추출한다.
- **결제 오케스트레이터(170)**: PortOne V2 빌링키 결제, 카카오페이 SID 정기결제, 애플/구글 IAP 구독 영수증 검증을 통합 처리한다.
- **알림 게이트웨이(180)**: 알림톡 API(예: ALIGO `alimtalk/send`)로 카카오 알림톡을 발송한다.
- **사전 데이터베이스(190)**: 질환별 평균 진료비(disease-cost-data), 질환별 치료/비용/생활영향(disease-treatments), 질환-후속질환 매핑(disease-risk-map), 예외질병(exception-diseases) 테이블.

### 2. 의료이력 수신 및 고지사항 분류 (도 2)

[0011] 설계사 클라이언트(110)는 피보험자의 진료 이력 PDF를 업로드하거나 또는 CODEF API를 통해 진료·약·건강검진 데이터를 수집한다. 애플리케이션 서버(120)는 PDF의 경우 PDF 파서(160)로 텍스트를 추출하고, 수집된 자료를 고지사항 분류 프롬프트(STEP1_ANALYSIS_PROMPT)로 LLM(140)에 전달한다.

[0012] 본 발명의 바람직한 실시예에 있어서, LLM(140)은 다음 20종의 카테고리 중 해당하는 항목별로 진료 이력을 자동 분류한다: `3months_visit`, `3months_medication`, `1year_hospitalization`, `1year_surgery`, `1year_visit`, `2year_hospitalization`~`5year_visit`, `5year_major_disease`, `ongoing_medication`.

[0013] **서버사이드 날짜 재검증**: LLM의 분류 결과에 포함된 각 진료 이력에 대하여, 애플리케이션 서버(120)는 `오늘 날짜 - 해당 진료 일자`를 실제 계산하여 LLM이 부여한 기간 카테고리를 보정한다. 이는 LLM의 시간 추론이 부정확할 수 있는 점을 보완하기 위함이다.

### 3. 위험도 리포트 생성 (도 3)

[0014] 고지사항 분류 결과로부터 피보험자 프로필(`PatientProfile`)을 추출한다. 이는 진단코드 집합, 진단명-코드 맵, 복용 약물 리스트, 전체 텍스트를 포함한다.

[0015] 상기 피보험자 프로필을 사전 구축된 질환-후속질환 매핑 테이블(DISEASE_RISK_MAP)과 매칭하여, 각 현재 질환에 대응하는 장래 위험 질환 목록(MatchedRisk)을 추출한다. 이때 매칭은 (i) 진단코드 직접 매칭, (ii) 복용 약물 매칭, (iii) 질환명 텍스트 매칭의 세 경로를 사용한다.

[0016] 상기 매칭 결과 및 피보험자 요약을 RISK_REPORT_PROMPT에 주입하여 LLM(140)을 호출하고, 그 응답으로부터 위험도 리포트(RiskReport)를 생성하여 데이터베이스(130)의 `analyses.risk_report` 컬럼에 저장한다.

### 4. 5년 후 인수 가능성 추정 (도 4)

[0017] 본 발명의 바람직한 실시예에 있어서, 인수 가능성 추정부(이하 "Underwriting Estimator")는 다음의 가중치를 적용한다:

**(a) 현재 진단코드 가중치** (DECLINE_RISK_CODES):

| 진단코드 prefix | 가중치 | 의미 |
|---|---|---|
| C (암 전체) | 40 | 암 진단/병력 |
| I21 | 35 | 급성심근경색 |
| I63 | 35 | 뇌경색/뇌졸중 |
| I50 | 25 | 심부전 |
| N18 | 30 | 만성콩팥병 |
| E11 | 15 | 제2형 당뇨 |
| I10 | 8 | 고혈압 |
| F32/F33 | 12/14 | 우울증/재발성 우울 |
| M05 | 18 | 류마티스관절염 |
| J44 | 18 | COPD |
| ... | ... | ... |

**(b) 카테고리별 가중치** (RISK_CATEGORY_WEIGHT):

| 카테고리 | 가중치 |
|---|---|
| 암 | 18 |
| 심혈관/뇌혈관 | 14 |
| 신장 | 12 |
| 대사/호흡기 | 8 |
| 정신/신경 | 6 |
| 근골격/소화기 | 4 |

**(c) 위험수준 배율**: high=1.0, moderate=0.6, low=0.3

**(d) 복용 약물 가산**: 5종 이상 +8점, 3종 이상 +4점

**(e) 위험 플래그 가산**: high severity +10점, medium +4점

[0018] 전체 점수 S는 다음과 같이 산출된다:

```
S = Σ(current_code_weight)
  + Σ(top5_risk: category_weight × level_multiplier)
  + medication_bonus
  + Σ(risk_flag_score)

declineRiskScore = clamp(S, 0, 100)
```

[0019] 산출된 점수에 기반하여 outcome을 결정한다:
- `declineRiskScore ≥ 60` → `likely_decline` (인수거절 가능성 높음)
- `30 ≤ declineRiskScore < 60` → `conditional` (조건부 인수)
- `declineRiskScore < 30` → `likely_accept` (인수가능성 높음)

[0020] 보험료 멀티플라이어는 `1.0 + min(0.5, declineRiskScore × 0.006)`로 산출한다. 점수 1점당 약 0.6%의 보험료 인상을 반영하며, 최대 +50%로 상한을 둔다.

### 5. 3시나리오 산출 (도 5)

[0021] 각 위험 질병에 대하여 다음과 같이 세 시나리오의 자기부담금을 산출한다:

```
totalNeed = 본인부담금 + 생활비손실
         = coveredSelfPay + uncoveredCost + monthlyLivingCost × treatmentMonths

[시나리오 A] 지금 보완
  A_payout = currentInsurancePayout + proposedInsurancePayout
  A_outOfPocket = max(0, totalNeed - A_payout)

[시나리오 B] 5년 후 가입 시도
  coverageRate = outcome 매핑 (accept 0.85, conditional 0.50, decline 0.10)
  if (질병이 예상 면책항목과 매칭) coverageRate ×= 0.3
  B_payout = currentInsurancePayout + proposedInsurancePayout × coverageRate
  B_outOfPocket = max(0, totalNeed - B_payout)

[시나리오 C] 미조치
  C_payout = currentInsurancePayout
  C_outOfPocket = max(0, totalNeed - C_payout)
```

[0022] 월 보험료는 시나리오별로 다음과 같이 산출된다:
- A: 설계사 입력값 그대로 (additionalMonthlyPremium)
- B: additionalMonthlyPremium × expectedPremiumMultiplier
- C: 0원

### 6. 시나리오 설명 자연어 생성 (도 6)

[0023] 산출된 시나리오 금액, 인수 추정, 질병별 치료/비용근거/생활영향 데이터를 FUTURE_ME_SCENARIOS_PROMPT에 주입하여 LLM(140)을 호출한다. LLM은 다음 JSON 구조로 응답한다:

```json
{
  "customerIntro": "고객용 리포트 도입 안내문",
  "diseaseNarratives": {
    "<diseaseCode>": {
      "whyRiskyForCustomer": "이 환자가 왜 이 질병에 위험한지",
      "riskInPlainLanguage": "상대위험도의 자연어 표현"
    }
  },
  "scenarios": {
    "A": { "title", "headline", "body", "highlights": [...] },
    "B": { ... },
    "C": { ... }
  },
  "keyTakeaways": ["핵심 결론 1", "결론 2", "결론 3"]
}
```

[0024] LLM 호출 실패 시에도 서비스가 중단되지 않도록, 시스템은 `fallbackRiskInPlainLanguage(rr)` 함수를 통해 상대위험도 숫자를 자연어로 변환하는 기본 메시지를 제공한다.

### 7. HMAC 토큰 공유 및 알림톡 전송 (도 7)

[0025] 설계사가 리포트를 고객에게 전달하기 위하여 "카톡으로 보내기"를 선택하면, 서버는 다음 페이로드를 HMAC-SHA256으로 서명한 토큰을 발급한다:

```
payload = { a: analysisId, i: issuedAt, e: expiresAt, u: issuerUserId }
body = base64url(JSON.stringify(payload))
sig  = base64url(HMAC_SHA256(secret, body))
token = body + "." + sig
```

[0026] 발급된 토큰을 파라미터로 포함하는 URL(`https://<host>/share/future-me/<token>`)을 생성하고, 이를 알림톡 템플릿의 웹링크 버튼 URL로 주입하여 ALIGO `alimtalk/send` API를 호출한다. 토큰 검증 시 `crypto.timingSafeEqual`을 사용하여 타이밍 공격을 방지한다.

[0027] 알림톡은 카카오 사전 승인 템플릿에 기반하여 발송되며, 본 발명의 실시예에서는 두 가지 템플릿을 사용한다: (i) 링크 전송형(template A) — 리포트 도착 알림과 공유 링크 버튼을 포함; (ii) 요약 전송형(template D) — 시나리오별 자기부담금 수치를 본문에 포함.

### 8. 추가 실시예

[0028] 본 발명은 정기결제 수단으로 다음의 결제 채널을 병행 지원할 수 있다: (i) PortOne V2 기반 KG이니시스 빌링키 신용카드 정기결제, (ii) 카카오페이 SID 정기결제, (iii) Apple/Google 인앱결제(IAP). 결제 수단별로 `subscriptions.payment_provider` 필드에 구분자를 저장하고, Vercel Cron 기반 일일 갱신 배치(`/api/cron/renew-subscriptions`)에서 제공자별 API를 호출하여 갱신 결제를 수행한다.

[0029] 본 발명은 서브-도메인 모듈로서 다음을 추가로 포함할 수 있다:
- 가입 가능 상품 매칭(STEP 2): 고지사항 분류 결과를 사전 구축된 보험상품 매트릭스 및 예외질환 DB와 대조하여 상품별 O/△/X 및 근거를 판정한다.
- 보험금 청구 가능성 분석(STEP 3): 진료 이력과 약관 특약을 대조하여 청구 가능 항목을 도출한다.
- 가상 사고 영수증(STEP 4): 특정 질환의 예상 진료비와 현재 보험 보장을 비교한다.

### 9. 하드웨어/소프트웨어 실행 환경

[0030] 본 발명의 방법은 범용 컴퓨터, 서버, 클라우드 컴퓨팅 환경에서 실행 가능하며, 예시적으로 다음의 환경에서 구현될 수 있다:

- 실행 환경: Node.js 20.x 이상
- 프레임워크: Next.js 16.x (App Router)
- 런타임: Vercel 서버리스 함수 또는 자체 호스팅 Node.js 서버
- 데이터베이스: PostgreSQL 14 이상 (예: Supabase)
- LLM: 클라우드 LLM API (예: Anthropic Claude, OpenAI GPT-4/5 등)
- 모바일 셸: Capacitor 8 기반 iOS/Android 네이티브 래퍼 (선택)

[0031] 본 발명의 방법을 구현하는 코드는 컴퓨터 판독 가능한 기록매체에 저장될 수 있으며, 해당 기록매체에는 마그네틱 저장매체, 광학 저장매체, 반도체 저장매체 등이 포함된다.
